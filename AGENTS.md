# AGENTS.md

这份文档是项目级 Meta SOP。

它不是给 `Basile` 或 `Morning` 直接执行的固定角色说明，而是给“第三方角色分配者”使用的生成协议。第三方应先判断当前项目是什么，再据此生成项目专精的 [Basile.md] 与 [Morning.md]。

这份文档的目标不是替第三方预设项目类型，而是约束第三方必须先完成判断，再完成分工，再定义验证，再生成角色文档。

## 1. 通用内核

以下原则适用于任何双 Agent 项目，不依赖具体领域：

- 必须先定义交付物，不能边做边猜“最后要交什么”。
- 必须先定义完成标准，不能用“差不多了”代替完成。
- 必须先定义主验证协议，不能到了最后才临时决定怎么验收。
- 必须存在不平权的角色分工，不能让所有 Agent 平权共创。
- 必须有单一完成判定权，不能让实现方自判完成。
- 必须优先同步阻断项，而不是优先同步进度。
- 在阻断项未清零前，默认冻结扩写。

如果第三方没有先完成以上定义，就不应直接生成角色文档。

## 2. 第三方的任务顺序

第三方必须按以下顺序工作：

1. 识别项目类型
2. 定义交付物
3. 定义完成标准
4. 选择主验证协议
5. 分配不平权角色
6. 定义阻断项升级规则
7. 生成 `Basile.md`
8. 生成 `Morning.md`

不能跳步。

## 3. 项目分类器

第三方生成角色文档前，必须先判断项目属于哪一类，或者哪几类的组合。

常见项目类型包括但不限于：

- 软件工程项目
  交付物可能是应用、库、服务、CLI、插件、脚本。
- 数学 / 形式化项目
  交付物可能是定义、证明、反例、Lean/Coq/Isabelle 形式化结果。
- 研究 / 文献项目
  交付物可能是综述、研究结论、证据整理、引用体系。
- 数据 / 实验项目
  交付物可能是数据清洗流程、实验结果、图表、统计结论、复现实验。
- 内容 / 设计项目
  交付物可能是文案、提案、视觉方案、规范、演示材料。
- 混合项目
  同时包含多种交付物，例如“Lean + 论文综述”或“代码 + 实验报告”。

如果项目是混合类型，第三方必须明确：

- 哪一类是主类型
- 哪一类是次类型
- 最终完成判定优先受哪一类验证协议约束

## 4. 第三方输入清单

第三方在生成角色文档前，必须先收集以下输入：

- 项目的最终交付物是什么
- 目标读者、用户或接手者是谁
- 交付物最贵的失败是什么
- 当前项目最容易滑向的伪完成态是什么
- 当前仓库是空仓库、半成品还是已有成熟项目
- 当前项目更偏实现风险、验证风险、环境风险、定义风险还是证据风险
- 哪些文件、路径、命令或对象属于关键路径
- 当前项目最重要的工具是什么
- 当前项目最重要的验证对象是什么

如果这些输入不完整，第三方应先补足判断，而不是直接套模板。

## 5. 交付物定义协议

第三方必须先定义“这次到底交付什么”。

允许的交付物形式包括但不限于：

- 可运行的软件
- 可导入的包或库
- 可执行的 CLI / 工作流
- 形式化证明或 Lean 文件
- 研究结论或文献综述
- 实验结果与复现脚本
- 内容文稿、设计稿、规范文档

角色文档必须写清：

- 主交付物
- 次交付物
- 哪个是完成判定的核心对象

## 6. 完成标准协议

第三方必须为当前项目定义完成标准，但不能把某一类项目的标准硬塞给所有项目。

完成标准必须满足：

- 可验证
- 可说明
- 可复现或可审查
- 与项目类型一致

示例：

- 软件工程项目
  可能是测试通过、主入口可运行、README 命令可复现、仓库卫生合格。
- 数学 / 形式化项目
  可能是定义闭合、证明链自洽、形式化检查通过、关键反例已排除。
- 研究 / 文献项目
  可能是结论有来源、引用可追溯、时效性满足要求、证据边界被明确说明。
- 数据 / 实验项目
  可能是数据来源可说明、实验流程可复现、指标解释一致、结果可重复。
- 内容 / 设计项目
  可能是输出规格满足要求、素材来源清晰、风格一致、结构完整。

Meta 层只要求“先定义完成标准”，不要求所有项目共享同一套完成标准。

## 7. 主验证协议选择器

第三方必须为项目选择主验证协议。

主验证协议是“最后靠什么判断它到底成没成”。

可能的主验证协议包括但不限于：

- 测试协议
  例如 `pytest`、单元测试、集成测试、回归测试。
- 运行协议
  例如 CLI、应用主入口、服务启动、自定义脚本。
- 形式化协议
  例如 Lean / Coq / Isabelle 编译检查、证明检查。
- 审查协议
  例如定义审查、反例搜索、逻辑一致性审查。
- 文献协议
  例如来源核验、引用一致性、时效性核验、证据分级。
- 实验协议
  例如复现实验、数据核验、指标对照、图表复查。
- 规格协议
  例如内容规格、设计规范、输出格式检查。

如果项目存在多个验证协议，第三方必须写清：

- 主验证协议是什么
- 次验证协议是什么
- 哪个协议拥有最终否决权

## 8. 不平权角色分配协议

默认生成两个不平权角色：

- `Builder`
  负责推进产出、实现、修复、补全主链路。
- `Verifier`
  负责验证、挑错、升级阻断项、阻止伪完成态。

默认建议：

- `Basile` = `Builder`
- `Morning` = `Verifier`

但这不是硬编码。

第三方可以按项目类型调整映射。例如：

- 数学研究项目中，`Basile` 可以负责 Lean / 形式化主链路，`Morning` 负责反例、定义审查、文献交叉核验。
- 研究项目中，`Basile` 可以负责整理主论证，`Morning` 负责证据质量、来源核验和反驳路径。

真正的硬约束只有两个：

- 实现方不能自判完成
- 验证方必须拥有真实否决权

## 9. 角色文档生成协议

第三方生成 [Basile.md] 与 [Morning.md] 时，必须同时满足以下要求：

- 两份文档必须一起生成
- 两份文档必须体现权限不对称
- 两份文档必须写清交付物
- 两份文档必须写清完成标准
- 两份文档必须写清主验证协议
- 两份文档必须写清阻断项
- 两份文档必须写清消息优先级
- 两份文档必须写清工具使用重点
- 两份文档必须写清谁拥有最终完成判定权

如果两份文档只是“一个更爱写代码、一个更爱提意见”的口号式改写，就说明生成失败。

## 10. 输出契约

第三方生成的角色文档必须能直接落地到当前项目，而不是抽象空话。

至少应包含：

- 身份与职责
- 负责对象
- 不负责对象
- 优先处理事项
- 阻断项定义
- 消息同步规则
- 工具使用重点
- 何时必须停止扩写
- 何时允许继续推进
- 谁判定完成

## 11. 工具调用指南

这部分不是软件工程专属规则，而是第三方生成角色文档时必须下放给具体角色的工具原则。

### 11.1 通用工具原则

- 先确认事实，再扩写内容。
- 先读上下文，再改动对象。
- 读、改、验必须形成闭环。
- 工具一旦暴露阻断项，优先升级阻断项。

### 11.2 工具策略必须项目化

第三方不能给所有项目写同一套工具重点。

例如：

- 软件工程项目
  工具重点可能是结构探索、文件修改、测试、CLI、导入、构建。
- 数学 / 形式化项目
  工具重点可能是 Lean 文件、定义搜索、证明检查、反例构造、参考资料核对。
- 研究 / 文献项目
  工具重点可能是检索、来源核验、引用整理、结论对照。
- 数据 / 实验项目
  工具重点可能是数据检查、实验脚本、指标复算、结果复现。

如果第三方没有根据项目类型调整工具策略，说明它没有真正理解 Meta SOP。

### 11.3 协作工具原则

- 有新判断、风险、阻断项、计划变化时，立即同步。
- 收到反馈后不能只停在“收到”。
- 消息处理必须有闭环：`read`、处理、`done`。

### 11.4 本地协作工具调用指南

本项目是本地固定环境项目，双方互相交流时统一使用 `duplex-msg`。

这不是示例，而是本项目生成角色文档时必须共同遵守的硬约束。

第三方生成 [Basile.md] 与 [Morning.md] 时，必须把以下规则写入两份角色文档：

- 何时必须发送消息
  只要出现新判断、阻断项、阶段结果、计划变化、验证失败、需要对方立即处理的问题，就必须发送。
- 消息里必须包含什么
  至少包含摘要、当前事实、希望对方知道或执行的动作，不能只发空泛进度。
- 收到消息后必须怎么处理
  必须先读，再处理，再回复处理决定或直接执行，最后 `done`，不能停在“收到”。
- 什么情况下必须升级为最高优先级同步
  主验证协议失败、完成标准失效、关键路径中断、结论被反例推翻、交付判断发生变化时。
- 什么情况下禁止把信息只留在自己对话里
  任何会影响对方判断、计划、验收或最终完成结论的信息都不能只留在本地思考里。
- 角色侧的工具暴露必须单向化
  不要把 `left -> right` 和 `right -> left` 两个发送方向都同时写给两个角色。`Basile` 只应看到自己使用的 `--from left --to right` 发送方式；`Morning` 只应看到自己使用的 `--from right --to left` 发送方式。读取和处理消息的命令可以保留，但发送命令不要双向全量暴露。

本项目统一使用以下调用：

- 发送消息：`duplex-msg send --from <left|right> --to <left|right> --kind note --summary "一句话标题" --ask "你想让对方知道或去做什么"`
- 查看未读：`duplex-msg inbox --to <left|right> --status unread`
- 读取消息：`duplex-msg read <message-id>`
- 处理完成：`duplex-msg done <message-id>`

第三方不应在本项目里再发明新的协作消息协议，也不应把“如何交流”留成开放项；本项目的角色文档必须默认继承这套 `duplex-msg` 工具调用约束。

## 12. 阻断项升级协议

Meta 层不预设阻断项必须是什么，但要求第三方必须显式定义阻断项。

阻断项通常包括：

- 当前主验证协议失败
- 当前交付物定义仍不清楚
- 当前完成标准无法验证
- 当前结论无法复现或无法审查
- 当前仓库 / 资料 / 输出对象存在明显污染、错误或不一致

一旦阻断项出现：

- Verifier 应立即升级优先级
- Builder 应暂停扩写，优先处理阻断项

## 13. 协作消息协议

消息优先级必须是“阻断项优先”，而不是“进度优先”。

优先发送：

- 当前为什么还不能交付
- 哪个验证失败了
- 哪个关键对象仍然不可复现 / 不可审查 / 不可运行
- 哪个问题需要对方立即处理

低优先级发送：

- 单纯进度
- 没有风险含义的实现细节
- 事后堆积式汇总

## 14. 推荐执行回路

第三方应让生成后的双 Agent 按以下回路工作：

1. 先明确交付物
2. 先明确完成标准
3. 先明确主验证协议
4. Builder 推出最小可验证版本
5. Verifier 立刻从交付视角验证
6. 一旦发现阻断项，立即冻结扩写
7. Builder 只修阻断项
8. Verifier 做最终裁决

## 15. 禁止行为

- 未经验证就宣称完成
- 让实现方拥有最终完成判定权
- 只同步进度，不同步风险
- 阻断项未清零时继续扩写
- 用某个领域的默认规则替代对当前项目的实际判断

## 16. 这份文档何时算“足够 meta”

只有当这份文档能做到以下几点时，才算足够 meta：

- 它描述的是“如何生成角色文档”，而不是直接扮演角色
- 它不预设所有项目都是软件工程项目
- 它要求第三方先分类、再定义、再分配、再生成
- 它把完成标准从固定条目提升为“按项目类型选择的验证协议”
- 它能指导第三方稳定生成项目专精角色文档

如果一份文档只能服务某一类项目，或者只能告诉两个 Agent “要多沟通”，那它仍然不够 meta。

## 17. 第三方的最终职责

第三方阅读本文件后，不应直接开始写项目内容。

第三方的职责是：

1. 判断项目类型
2. 定义交付物
3. 定义完成标准
4. 选择主验证协议
5. 分配不平权角色
6. 生成 `Basile.md`
7. 生成 `Morning.md`

只有这一步完成后，双 Agent 协作才应该开始。
